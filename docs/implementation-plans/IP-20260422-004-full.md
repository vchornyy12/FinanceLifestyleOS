# Implementation Plan: Income / Expense / Transfer — Manual Entry

> **For agentic workers:** Use superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Let the user record income and transfers alongside expenses via manual entry, and show real income / expense / net metrics on the dashboard. Banking integration deferred until later.

**Tech Stack:** Supabase (Postgres + RLS) · Next.js 16 App Router · Server Actions · Zod · Expo Router · NativeWind

---

## Key Technical Decisions

| Decision | Rationale |
|----------|-----------|
| Option A — `transaction_type` enum (`expense` / `income` / `transfer`) | Explicit, easy to aggregate, works for future `'refund'` / `'adjustment'` values |
| `amount` stays positive-only (`CHECK (amount > 0)`) | Sign is implied by `type`; avoids ambiguity on transfers |
| Transfers stored as a single row with `from_account` / `to_account` text fields | No `accounts` table yet — strings become FKs in Phase 4 |
| Transfers excluded from income & expense aggregates | A transfer is neither income nor spend; including it double-counts |
| Backfill existing rows as `'expense'` | All 12 seeded categories + every existing row are expense-shaped |
| Category remains optional for all types | Income/transfer often has no category; don't force one |

## Risk Factors

| Risk | Mitigation |
|------|------------|
| Existing rows break when `type` becomes NOT NULL | Migration adds column with `DEFAULT 'expense'` then drops the default — existing data is backfilled automatically |
| Transfer double-counting if someone writes `SUM(amount)` | Aggregate helpers always filter by type; document convention in `lib/supabase/queries/transactions.ts` |
| Form complexity: merchant label wrong for income/transfer | Dynamic label — "Merchant" for expense, "Source" for income, hidden for transfer (replaced by from/to) |
| Mobile and web drift out of sync | Both apps updated in the same IP; duplicated Zod schema until `packages/shared` is extracted (future) |

## Codebase Conventions (existing — follow these)

| Convention | Pattern |
|------------|---------|
| Server action errors | `{ fieldErrors }` for validation, `{ error: '...' }` for failures — never raw Supabase error messages |
| Server action auth | Always `supabase.auth.getUser()` — never trust client user_id |

### Task 1: DB Migration 004 — transaction_type enum + transfer columns

**Files:**
- Create: `supabase/migrations/004_transaction_type.sql`
- Create: `supabase/migrations/004_transaction_type_down.sql`

- [ ] **Step 1: Write migration SQL**

```sql
-- supabase/migrations/004_transaction_type.sql

-- 1. Enum for the three transaction types
CREATE TYPE public.transaction_type AS ENUM ('expense', 'income', 'transfer');

-- 2. Add type column, backfilled to 'expense' for existing rows
ALTER TABLE public.transactions
  ADD COLUMN type public.transaction_type NOT NULL DEFAULT 'expense';

-- Remove default so future inserts are explicit about intent
ALTER TABLE public.transactions
  ALTER COLUMN type DROP DEFAULT;

-- 3. Transfer endpoint columns (free-text until accounts table lands in Phase 4)
ALTER TABLE public.transactions
  ADD COLUMN from_account TEXT,
  ADD COLUMN to_account   TEXT;

-- 4. Data integrity
ALTER TABLE public.transactions
  ADD CONSTRAINT transactions_amount_positive CHECK (amount > 0);

ALTER TABLE public.transactions
  ADD CONSTRAINT transactions_transfer_endpoints CHECK (
    (type = 'transfer' AND from_account IS NOT NULL AND to_account IS NOT NULL)
    OR (type <> 'transfer' AND from_account IS NULL AND to_account IS NULL)
  );

-- 5. Index to speed up type-filtered aggregates on the dashboard
CREATE INDEX idx_transactions_user_type_date
  ON public.transactions (user_id, type, date DESC);
```

- [ ] **Step 2: Write rollback SQL**

```sql
-- supabase/migrations/004_transaction_type_down.sql
DROP INDEX IF EXISTS idx_transactions_user_type_date;
ALTER TABLE public.transactions DROP CONSTRAINT IF EXISTS transactions_transfer_endpoints;
ALTER TABLE public.transactions DROP CONSTRAINT IF EXISTS transactions_amount_positive;
ALTER TABLE public.transactions DROP COLUMN IF EXISTS to_account;
ALTER TABLE public.transactions DROP COLUMN IF EXISTS from_account;
ALTER TABLE public.transactions DROP COLUMN IF EXISTS type;
DROP TYPE IF EXISTS public.transaction_type;
```

- [ ] **Step 3: Apply migration** via Supabase MCP or dashboard SQL editor. Verify `\d public.transactions` shows new columns and existing rows have `type = 'expense'`.

- [ ] **Step 4: Verify RLS still applies** — no policy changes needed (policies are row-level and type-agnostic).

- [ ] **Step 2: Create the shared Zod schema** in `apps/web/lib/schemas/transaction.ts`:

```typescript
import { z } from 'zod'

export const TransactionTypeEnum = z.enum(['expense', 'income', 'transfer'])

export const TransactionSchema = z.object({
  type: TransactionTypeEnum,
  merchant: z.string().min(1).max(200),              // required for expense/income; for transfer, store "—" or omit
  amount: z.string().regex(/^\d+(\.\d{1,2})?$/),
  category_id: z.string().uuid().nullable().optional(),
  date: z.string().regex(/^\d{4}-\d{2}-\d{2}$/),
  note: z.string().max(500).optional(),
  from_account: z.string().min(1).max(100).nullable().optional(),
  to_account:   z.string().min(1).max(100).nullable().optional(),
}).superRefine((data, ctx) => {
  if (data.type === 'transfer') {
    if (!data.from_account) ctx.addIssue({ code: 'custom', path: ['from_account'], message: 'Required for transfers' })
    if (!data.to_account)   ctx.addIssue({ code: 'custom', path: ['to_account'],   message: 'Required for transfers' })
  } else {
    if (data.from_account) ctx.addIssue({ code: 'custom', path: ['from_account'], message: 'Only allowed on transfers' })
    if (data.to_account)   ctx.addIssue({ code: 'custom', path: ['to_account'],   message: 'Only allowed on transfers' })
  }
})
```

- [ ] **Step 3: Mobile mirrors this schema** — copy to `apps/mobile/lib/schemas/transaction.ts`. Note: duplication is intentional until `packages/shared` is extracted.
